Getting Started

The Difference between Git and other VCS

Most other systems store information as a list of file-based changes. These other systems think of the information they store as a set of files and the changes made to each file over time (this is commonly described as delta-based version control).

Git thinks of its data more like a series of snapshots of a miniature filesystem. With Git, every time you commit, or save the state of your project, Git basically takes a picture of what all your files look like at that moment and stores a reference to that snapshot. To be efficient, if files have not changed, Git doesn’t store the file again, just a link to the previous identical file it has already stored. Git thinks about its data more like a stream of snapshots.

The Three States

Git has three main states that your files can reside in: modified, staged, and commited.

  • Modified means that you have changed the file but have not commited it to your database yet.
  • Staged means that you have marked a modified file in its current version to go into your next commit snapshot.
  • Commited means that the data is safely stored in your local database.

This leads to thress main sections of a Git project: the working tree, the staging tree, and the Git directory.

The working tree is a single checkout of one version of the project. These files are pulled out of the compressed database in the Git directory and placed on disk for you to use or modify.
The staging area is a file, generally contained in your Git directory, that stores information about what will go into your next commit. Its technical name in Git parlance is the “index”, but the phrase “staging area” works just as well.
The Git directory is where Git stores the metadata and object database for your project. This is the most important part of Git, and it is what is copied when you clone a repository from another computer.

The basic Git workflow

The basic Git workflow goes something like this:

  1. You modified files in your working tree.
  2. You selectively stage just those changes you want to be part of your next commit, which adds only those changes to the staging area.
  3. You do a commit, which takes the files as they are in the staging area and stores that snapshot permanent to your Git directory.

git config

a tool called git config that lets you get and set configuration variables that control all aspects of how Git looks and operates. These variables can be stored in three different places:

  1. [path]/etc/gitconfig file: contains values applied to every user on the system and all their repositories. This is a system configuration file. If you pass the option --system to git config, it reads and writes from this file specifically.
  2. ~/.gitconfig or ~/.config/git/config file: Values specific personally to you, the user. You can make make Git read and write to this file specifically by passing the --global option, and this affects all of the repositories you work with on your system.
  3. config file in the Git directory of whatever repository you’re currently using: Specific to that single repository. You can force Git to read from and write to this file with the –local option, but that is in fact the default. Unsurprisingly, you need to be located somewhere in a Git repository for this option to work properly.

Each level overrides values in the previous level, so values in .git/config trump those in [path]/etc/gitconfig.

Default Branch Name

git config --global init.defaultBranch main

Checking Your Settings

show all configurations

git config --list

show the specific value

git config <key>

Git Basic

Clone an Existing Repository

If you want to clone the repository into a directory named something other than libgit2, you can specify the new directory name as an additional argument:

git clone https://github.com/libgit2/libgit2 mylibgit

The git clone command automatically sets up your local master branch to track the remote master branch (or whatever the default branch is called) on the server you cloned from.

Recording Changes to the Repository

the each file in working directory can be in one of two stages: tracked and untracked.

Tracked files are files that were in the last snapshot; they can be unmodified, modified, or staged.

Untracked files are everything else - any files in working directory that were not in your last snapshot and are not in your staging area. When you first clone a repository, all of your files will be tracked and unmodified because Git just checked them out and you haven’t edited anything.

checking the status of your file

The main tool you use to determine which files are in which state is the git status command.

git status

Tracking new files

In order to begin tracking a new file, you use the command git add.

git add file_name

Staging Modified Files

To staged the file that is tracked has been modified in the working directory but not yet staged. To stage it, you run the git add command.

git add file_name

git add is a multipurpose command – you use it to began tracking new files, to stage files, and to do other things like marking merge-conflicted files as resolved.

Short Status

If you run git status -s or git status --short you get a far more simplified output from the command:

git status -s

?? – new files that aren’t tracked have a ?? next to them

A – new files that have been added to the staging area have an A

M – modified files have an M

Ignoring Files

The rules for the patterns in the .gitignore file as follows:

  • Blank lines or lines starting with # are ignored.
  • Standard glob patterns work, and will be applied recursively throughout the entire working tree.
  • You can start patterns with a forward slash (/) to avoid recursivity.
  • You can end patterns with a forward slash (/) to specify a directory.
  • You can negate a pattern by starting it with an exclamation point (!).

An asterisk (*) matches zero or more characters; [abc] matches any character inside the brackets (in this case a, b, or c); a question mark (?) matches a single character; and brackets enclosing characters separated by a hyphen ([0-9]) matches any character between them (in this case 0 through 9). You can also use two asterisks to match nested directories; a/**/z would match a/z, a/b/z, a/b/c/z, and so on.

Viewing Your staged and Unstaged Changes

#This command only compares what is in your working directory with what is in your staging area.
git diff

#This command compares your staged changes to your last commit.
git diff --staged

#This command to see what you've staged so far (--staged and --cached are synonyms)
git diff --cached

Skipping the Staging Area

Adding the -a option to the git commit command makes Git automatically stage every file that is already tracked before doing the commit, letting you skip the git add part: 

git commit -a -m "xxxx"

Removing Files

If you simply remove the file from your working directory, it shows up under the “Changes not staged for commit” (that is, unstaged) area of your git status output.

#This command removes the file from your tracked files (more accurately, remove it from your staging area) and then commit. And it also removes the file from your working directory. The next time you commit, the file will be gone and no longer tracked.
git rm file_name

# -f option 
# If you modified the file or had already added it to the staging area, you must force the removal with -f option
git rm -f file_name

#--cached option
#The command will keep the file in your working tree but remove it from your staging area.
git rm --cached file_name

Viewing the Commit History

#-p or --patch shows the difference(the patch output) introduced in each commit.
git log -p 

#limit the number of log entries displayed, such as using -2 to show the last two entries
git log -p -2

# --stat option can be used to see some abbreviated stats for each commit.
# the --stat option prints below each commit entry a list of modified files, how may files were changed, and how many lines in those files were added and removed.
git log --stat

# --format allows you to specify your own log output format
git log --pretty=format:"%h - %an, %ar:%s"

Common options to git log as follows:

Limiting Log Output

  • --since and --until are the time-limiting options. For example:

    # this command gets the list of commits made in the last two weeks
git log --since=2.weeks

  • -author option allows you to filter on a specific author

  • -grep option lets you search for keywords in the commit messages.

  • -S option which takes a string and shows only those commits that changed the number of occurrences of that string.

    git log -S function_name

  • -- the path as a filter pass to git log

    This is always the last option and is generally preceded by double dashes(–) to separate the paths from the options:

    git log -- path/to/file

  • Options to limit the output of git log

Undoing Things

amend

  • git commit --amend

    When you commit too early and possibly forget to add some files, or you mess up your commit message. If you want to redo that commit, make the additional changes you forgot, stage them, and commit again using the --amend option.

    Only amend commits that are still local and have not been pushed somewhere.

    # This command takes your staging area and uses it for the commit
git commit --amend

# for example:
git commit -m 'Initial commit'
git add forgotten_file
git commit -amend

Unstaging a Staged File

git reset HEAD filename

Unmodifying a Modified File

  • git checkout -- file_name

    Any local changes you made to that file are gone.

    git checkout -- file_name

Undoing things with git restore

Unstaging a Staged File with git restore

git restore --staged <file>

Unmodifying a Modified File with git restore

Any local changes you made to that file are gone – Git just replaced that file with the last staged or commited version.

git restore <file>

Working with Rmotes

Showing Your Remotes

Git要求每个远程主机都必须指定一个主机名。git remote命令用于管理主机名。

# It lists the shortname of each remote handle you've specified. If you've cloned your respository, you should at least see origin. origin is the default name Git gives to the server you cloned from:
git remote

# This command shows you the URLs that Git has stored for the shortname to be used when reading and writing to that remote
git remote -v

Adding Remote Repositories

git remote add <主机名> <网址>

for example:

Delete Remote Repositories

git remote rm <主机名>

Rename Remote Repositories

git remote rename <原主机名> <新主机名>

Fetching and Pulling from Your Remotes

先行概念：

  • FETCH_HEAD：版本链接，记录在本地的一个文件中，指向目前已经从远程仓库取下来的分支的末端版本
  • commit-id：每次本地工作完成后，使用git commit操作来保存当前工作到本地的repo，此时会产生一个commit-id，这是一个能唯一标识一个版本的序列号。在使用git push后，这个序列号还会同步到远程仓库

  1. **git fetch ** – 不会自动合并

    将更新git remote 中所有的远程仓库所包含分支的最新的commit-id，并将其记录到.git/FETCH_HEAD 文件中

    #Step one:
#将某个远程主机的更新全部取回本地，默认情况下，取回所有分支的更新
git fetch <远程主机名>

# 取回特定分支的更新，指定分支名
git fetch <远程主机名> <分支名>

#A power move which fetches all registered remotes and their branches.
git fetch --all

#Step Two
# Using the git checkout command to checkout the newly downloaded remote branch. 
git checkout <分支名>

#Step Three
#We could create a new branch to develop from the latest remote content.
git checkout -b <新分支名>




#在本地新建一个tmp分支，并将远程origin仓库的master分支代码下载到本地temp分支
git fetch origin master:tmp
#来比较本地代码与刚刚从远程下载下来的代码的区别
git diff tmp
#合并temp分支到本地的master分支
git merge tmp
#如果不想保留temp分支 可以使用此步删除
git branch -d temp

  2. git pull – 自动合并

    git pull == git fetch + git merge

    首先，基于本地的FETCH_HEAD记录，比对本地的FETCH_HEAD记录与远程仓库的版本号，然后git fetch获取当前指向的远程分支的后续版本的数据，然后再利用git merge将其与本地的当前分支合并。

    #取回远程主机某个分支的更新，再与本地的指定分支合并
git pull <远程主机名> <远程分支名>:<本地分支名>

#for example
#将远程分支的next分支与本地的master分支合并
git pull origin next:master
#如果远程分支是与当前分支合并，则冒号后的部分可以省略(同理，取回origin/next分支，再与当前分支合并)
git pull origin next

Pushing to Your Remotes

git push <远程主机名> <本地分支名>:<远程分支名>

# for example
# If you want to push your master branch to your origin server
git push origin master

Inspecting a Remote

#To see a more information about a particular remote
#It lists the URL for the remote repository as well as the tracking branch information.
git remote show <remote>

手动建立追踪关系

#指定master分支追踪origin/next分支
git branch --set-upstream master origin/next

Renaming and Removing Remotes

#To change a remote's shortname
#this also changes all your remote-tracking branch names
git remote rename <old_remote_name> <new_remote_name>

#Remove a remote for some reason
#Once you delete the reference to a remote this way, all remote-tracking branches and configuration settings associated with that remote  are also deleted.
git remote remove <remote>

Tagging

Listing Tags

#The command lists the tags in alphabetical order
git tag

#Search for tags that match a particular pattern
git tag -l <pattern>

Creating Tags

Two types of tags: lightweight and annotated.

A lightweight tag is very much like a branch that doesn’t change — it’s just a pointer to a specific commit.

Annotated tags, however, are stored as full objects in the Git database. They’re checksummed; contain the tagger name, email, and date; have a tagging message; and can be signed and verified with GNU Privacy Guard (GPG).

Annotated Tags

# -m specifies a tagging message, which is sorted with the tag
git tag -a <version tags> -m <comment>
# for example
git tag -a v1.4 -m "my version 1.4"
git tag --annotate v1.4 -m "my version 1.4"


# See the tag data along with the commit that was tagged
git show <tag_name>
# for example
git show v1.4

Lightweight Tags

This is basically the commit checksum stored in a file – no other information is kept. To create a lightweight tag, don’t supply any of the -a, -s, or -m options, just provide a tag name:

git tag <tag_name>
# for example
git tag v1.4-lw

Tagging Latter

Tag commits after you’ve moved pass them.

Sharing Tags

By default, the git push command dosen’t transfer tags to remote servers. You will have to explicitly push tags to a shared server after you have created them. You can run git push origin <tagname>

git push origin <tagname>
# for example
git push origin v1.5

# If you have a lot of tags that you want to push up at once, you can also use the --tags option to git push command
git push origin --tags

# git push pushes both types of tags
# There is currently no option to push only lightweight tags, but if you use `git push <remote> --follow-tags` only annotated tags will be pushed to the remote.

Deleting Tags

We could remove lightweight tag above as follows:

  • Step One，but this does not remove the tag from any remote servers
    git tag -d 

    git tag -d v1.4-lw

  • Step Two: There are two common variations for deleting a tag from a remote server.

    • first variation: git push <remote> :refs/tags/<tagname>

      git push origin :refs/tags/v1.4-lw

    • second variation: git push origin --delete <tagname>

Checking out Tags

If you want to view the versions of files a tag is pointing to, you can do a git checkout of that tag.

git checkout <tagname>

If you want to make changes, you will generally want to create a branch:

git checkout -b <new branch name> <tag name>

Git Aliases

You can easily set up an alias for each command using git config. Here are a couple of examples you may want to set up:

#git co resprensent checkout
git config --global alisa.co checkout
git config --global alisa.br branch
git config --global alias.ci commit  
git config --global alias.st status

Git Branching

Branches in Nutshell

creating a new branch

# 创建新的分支 
# This creates a new pointer to the same commit you're currently on 
# It didn't switch to that branch.
git branch <new_branch_name>

# This command will shows you where the branch poniter are pointing.
git log --oneline --decorate

Git keeps a special pointer called HEAD. In Git, this is a pointer to the local branch you’re currently on.

Switching Branches

#To switch an exsiting branch, you run the git check command.
#This command moves HEAD to target_branch_name
git checkout <target_branch_name>

#for example
git checkout testing
vim test.rb
git commit -a -m "made a change"
# When the before command finished, your testing branch has moved forward, but your master branch still points to the commit you were on when you ran git checkout to switch branches.

#After you used 'git checkout master' and 'git log', you might wonder where the "testing" branch you just created went, as it would not appear in the output.
#`git log` will only show commit history below the branch you've checked out.

Tips

#creating a new branch and switching to it at the same time
git checkout -b <new branch name>

#From git version 2.23 onwards you can use git switch instead of git checkout to:
#switch to an exsiting branch
git switch testing-branch
#create a new branch and switch to it
git switch -c new-branch
git switch -create new-branch
#Return to your previously checked out branch
git switch -

Branch Management

git branch command can be used to create, delete branches. It also can be used to list your current branches:

git branch -v
git branch <new_branch_name>
git branch -d <branch_name>

Changing a branch name

# rename the branch locally
git branch --move <old_branch_name> <new_branch_name>

# To let others see the new branch name on the remote,push it:
git push --set-upstream origin <new_branch_name>

# To delte the old branch name on the remote
git push origin --delete <old_branch_name>

Changing the master branch name

# remane local `master` branch into `main` 
git branch --move master main

# push the main branch to the remote
git push --set-upstream origin main

#Now you have a few more tasks in front of you to complete the transition:
# Any projects that depend on this one will need to update their code and/or configuration.
# Update any test-runner configuration files.
# Adjust build and release scripts.
# Redirect settings on your repo host for things like the repo’s default branch, merge rules, and other things that match branch names.
# Update references to the old branch in documentation.
# Close or merge any pull requests that target the old branch.

# Delete the master branch
git push origin --delete master
